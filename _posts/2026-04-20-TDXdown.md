## Summary for TDXdown: Single-Stepping and Instruction Counting Attacks against Intel TDX

## Introduction
민감한 데이터를 클라우드에서 처리하면서도 서비스 제공자로부터 데이터를 보호할 수 있는 TEE가 널리 사용되고 있습니다. Intel TDX, AMD SEV, ARM CCA와 같은 최신 세대 TEE는 **VM 전체를 보호** 하여 최소한의 수정만으로 기존 클라우드 애플리케이션을 안전하게 실행할 수 있도록 합니다. 그러나 TCB에서 하이퍼바이저를 제거하면서, 시스템 전체를 제어하는 공격자를 상정하는 강력한 공격자 모델이 도입되었고, 이를 바탕으로 TEE를 대상으로 한 다양한 공격 연구들이 이어졌습니다. 페이지 테이블 조작이나 캐시, BTB 관찰을 통해 TEE 내부의 민감한 데이터를 유출할 수 있음이 알려졌으며, Intel SGX와 AMD SEV에서는 **인터럽트 기반의 single-stepping 공격** 도 연구되었습니다. Single-stepping 공격은 APIC 타이머를 조작하여 TEE를 명령어 하나씩 실행시킴으로써 내부 제어 흐름을 명령어 수준까지 추론할 수 있는 강력한 기법으로, 이에 대응하여 Intel은 SGX를 위한 방어 메커니즘인 **AEX-Notify** 를 도입하였습니다.

Intel TDX 역시 single-stepping 공격에 대한 방어 기법을 내장하고 있습니다. AEX-Notify와 달리 TEE 내부 소프트웨어에 의존하지 않고, **TDX security monitor인 TDX 모듈 내부** 에 방어 로직을 구현하였습니다. 이 방어 기법은 **single-stepping 시도를 감지하는 detection heuristic과, 탐지 시 활성화되는 prevention mode** 로 구성됩니다. 본 논문에서는 이 방어 기법을 체계적으로 분석하고, 각각 다른 취약점을 이용하는 두 가지 공격을 제시합니다.

## Background
### TDX
![t1](/assets/img/posts/papers/tdxdown_1.png) <br>

**Intel Trust Domain Extensions (TDX)** 는 VM 전체에 대한 TEE를 제공합니다. 보호되는 VM은 **Trusted Domain** 이라고 불리고 **TDX module** 은 새로 추가된 보호되는 메모리 영역에 존재하면서 디자인의 핵심을 맡습니다. 일반 VM과 다르게 하이퍼바이저가 TD를 관리하려면 TDX module의 **SEAMCALL API** 를 호출해야 합니다. 하이퍼바이저는 TD에 직접 진입하지 못하고, TD에서 제어 흐름을 반환할 때도 TDX module에 반환합니다. 따라서 TDX module은 **비신뢰 하이퍼바이저와 TD 사이에 trusted layer** 을 구성한다고 할 수 있습니다. 

TDX는 **Secure Arbitration Mode (SEAM)** 이라는 새로운 CPU모드를 추가하였는데, 이는 둘로 나뉘어 TDX module은 `SEAM VMX root mode`, TD는 `SEAM VMX non-root mode`에서 동작합니다. 물리 공격자로부터 TD 메모리를 보호할 수 있도록 **Multi Key Total Memory Encryption (MKTME)** 를 활용하여 메모리를 암호화합니다. MKTME는 **KeyID** 에 따라 서로 다른 암호화 키를 사용하며, KeyID는 물리 주소의 상위 비트에 인코딩됩니다. TDX는 KeyID 범위를 shared와 private으로 나누어, private KeyID는 SEAM 모드에서만 접근 가능하도록 제한합니다. 따라서 하이퍼바이저가 TD의 메모리 주소에 접근하더라도 올바른 KeyID를 사용할 수 없어 TD의 데이터를 읽을 수 없습니다. 추가로 SEAM 모드 밖에서의 접근 권한 기반 보호 메커니즘도 존재하는데, 보호된 메모리를 읽으면 고정된 패턴이 반환되어 ciphertext 사이드채널 공격을 방어하고, 쓰기 시도 시 해당 **메모리 위치가 오염** 되어 TD가 다음에 접근할 때 치명적 오류가 발생합니다.

### TDX Control Data Structures
TD를 나타내는 구조체들은 TDX module에 의해 관리가 됩니다. 그 중 하나는 **Trust Domain Virtual Processor State (TDVPS)** 로, TD의 각 가상 CPU의 상태를 나타냅니다. 구조체들이 사용할 메모리 공간은 하이퍼바이저에 의해 초기에 할당된 후 TDX module로 전달되어 private MKTME KeyID를 통해 암호화가 됩니다. 따라서 하이퍼바이저는 구조체에 접근하려면 TDVPS 메모리 주소를 가지고 TDX module의 API를 사용해야 합니다.

### Cache Attacks on Intel TDX
MKTME는 캐시 부채널 공격의 대상이 될 수 있습니다. MKTME **KeyID가 물리 주소 비트에 인코딩** 되어 있기 때문에 캐시 태그의 일부를 구성합니다. 따라서 하이퍼바이저가 다른 KeyID로 TD 메모리에 접근하면 캐시 입장에서는 다른 주소로 취급이 됩니다. 이때 캐시 일관성 메커니즘으로 인하여 다른 KeyID로 같은 주소에 접근하면, 기존에 캐시에 있던 항목을 자동으로 flush하고 새로 로드를 하게 되는데, 이 동작 때문에 하이퍼바이저가 **다른 KeyID로 메모리에 접근해도 TD의 캐시 라인을 축출** 할 수 있게 됩니다. 따라서 KeyID 기반 Flush+Reload 공격이 가능해집니다. 

### Instruction Counting Attacks
Instruction Counting은 single-stepping 기반의 부채널 공격으로, TEE 내부의 세밀한 제어 흐름을 추론할 때 사용됩니다. 공격자는 실행되고 있는 바이너리를 알고 있다고 가정합니다. 페이지 폴트 기반 부채널만으로는 페이지 단위의 제어 흐름 정보만 얻을 수 있지만, 이를 single-stepping과 결합하면 victim의 제어 흐름을 명령어 단위까지 재구성할 수 있습니다. 심지어 길이가 같은 두 코드 브랜치가 서로 다른 시점에 메모리에 접근하는 경우도 탐지할 수 있습니다. 이러한 공격을 방지하려면 보안에 민감한 코드는 실행 경로와 메모리 접근 모두 비밀 데이터에 의존하지 않는 **constant-time paradigm** 에 따라 설계되어야 합니다.

### Elliptic Curve Digital Signature Algorithm

ECDSA는 타원 곡선 기반의 디지털 서명 알고리즘입니다. 서명을 위해 유한체 $\mathbb{F}_p$ 위의 타원 곡선 $E(\mathbb{F}_p)$와 위수 $n$의 생성원 $G$를 선택합니다. 서명자는 장기 비밀키 $d \in [1, n-1]$와 공개키 $Q = d \cdot G$를 생성합니다. 서명마다 비밀 nonce $k \in [1, n-1]$를 새로 생성하며, 메시지 해시 $h$에 대한 서명 $(r, s)$는 다음과 같이 계산됩니다.

$$r = x_r \mod n, \quad (x_r, y_r) = k \cdot G$$

$$s \equiv k^{-1}(h + r \cdot d) \mod n$$

ECDSA의 보안은 nonce $k$의 무작위성에 크게 의존합니다. $k$에 아주 작은 편향만 있어도 장기 비밀키 $d$를 복원할 수 있으며, 복원 방법으로는 격자 기반 기법(LLL, BKZ)과 Bleichenbacher의 푸리에 분석 기법이 있습니다.

### Attacker Model
논문에서는 기본적인 TEE 공격자 모델을 그대로 적용하여, 공격자가 관리자 권한을 갖고 시스템을 제어할 수 있는 상태에서 TEE 내부의 프로그램을 공격하는 상황을 전제합니다. 공격자는 페이지 폴트 기반 controlled-channel 공격, APIC 타이머와 프로세서 주파수 등을 조작할 수 있습니다. 실험을 위해 하드웨어 캐시 프리페처, Intel SpeedStep, 하드웨어 P-state를 비활성화하였으며, 물리적 접근은 가정하지 않습니다.

## TDX Single-Stepping Countermeasure

![t2](/assets/img/posts/papers/tdxdown_2.jpg) <br>

### TDX Interrupt Architecture
Intel SGX와 AMD-SEV 모두 single-stepping 공격자는 APIC 타이머와 같은 외부 인터럽트가 발생하면 TEE 실행이 중단되고 제어 흐름이 OS나 하이퍼바이저로 넘어오는 것을 활용합니다. 하지만 Intel TDX는 동작 방식에 차이가 있습니다. 공격자가 제어하는 하이퍼바이저가 바로 TD로 진입하거나, TD에서 나갈 때 바로 하이퍼바이저로 제어 흐름이 넘어가도록 할 수 없고, **TDX module** 이라는 신뢰 중간 계층이 존재합니다. 다만 TDX module은 하이퍼바이저를 대체하는 것이 아니기 때문에, 가상화 동작을 위해 결국 하이퍼바이저에게 TD 인터럽트 발생을 알려야 하는 경우가 생깁니다. 인터럽트 기반 single-stepping 공격을 방지하기 위해, TDX module은 특정 인터럽트 패턴에서 공격이 감지되면 **single-stepping prevention mode** 를 통해 랜덤한 수의 명령어를 추가로 실행한 뒤 하이퍼바이저에게 제어를 넘깁니다.

### Single-Stepping Detection
TD가 인터럽트로 인해 중단되었을 때마다 TDX module은 두 가지 속성을 통해 인터럽트의 정상 여부를 판단합니다. 첫 번째는 **TD의 RIP가 진행된 바이트 개수** 입니다. 만약 RIP가 가장 긴 x86 명령어 길이의 두배 이상 진행되었다면 정상 흐름으로 간주합니다. 두 번째는 **TDX module을 통해 TD에 진입한 후 소요된 시간** 입니다. 소요 시간을 측정하기 위해 `rdtsc` 타임스탬프를 TD 진입 전/후 기록하여 충분한 시간이 지났는지 비교합니다. TDX module 1.5 버전에서의 기준값은 4096 rdtsc increment입니다. 

위 두 가지 조건 중 하나 이상을 만족하면 정상 흐름으로 간주되고, 두 조건 모두 만족하지 못하면 prevention mode가 작동합니다.

### Prevention mode
Prevention mode의 핵심은 하이퍼바이저에게 인터럽트 발생을 전달하기 전에 `PmSteps`회만큼 랜덤한 수의 명령어를 실행하는 것입니다. Single-stepping 시도가 감지되면 TDX module은 먼저 하이퍼바이저가 prevention mode 실행에 개입하지 못하도록 인터럽트를 막습니다. TDX module은 SEAM VMX root mode에서 동작하기에 NMI, SMI와 같은 인터럽트도 전부 해당이 됩니다. 이후 TDX module은 하나의 명령어만 실행하게 하는 `Monitor Trap Flag`를 갖고 TD에 진입하여 `PmSteps`회만큼 명령어를 실행합니다. 

## Single-Stepping Trust Domains
### Attack Primitive Description
공격자의 목표는 TDX module이 측정한 rdtsc 값을 $t_d \ge 4096$이 되도록 조작하는 것입니다. 최신 Intel CPU에서 `rdtsc` 타임스탬프 카운터는 CPU 주파수에 관계없이 일정한 주파수에 따라 증가합니다. 이 점을 이용하여 하이퍼바이저가 TD와 TDX module이 실행되고 있는 코어의 주파수를 낮추면 TD에서 정확히 하나의 명령어만 실행이 되도록 하게 하면서도 TDX module이 측정한 rdtsc 카운터 값이 4096이 넘어가도록 설정할 수 있습니다. 

### Reliable Single-Stepping
[SGX-Step](https://jihooniab1.github.io/posts/SGX-step/), [AEX-Notify](https://jihooniab1.github.io/posts/AEX-Notify/)에서 설명된 것처럼 본 연구에서는 TD의 TLB를 flush 하여 첫번째 명령어의 실행 시간을 늘리고 single-stepping window의 크기를 증가시켰습니다. 

Multi-stepping을 방지하기 위해 윈도우의 끝보다 시작 부분에 가깝게 타이머가 도착하게 설명하면 zero-step으로 이어질 수 있지만, TD가 실행 중인 코드 페이지에 KeyID 기반 Flush+Reload 공격을 사용하여 실행 여부를 알아낼 수 있습니다. 하이퍼바이저가 TD에 진입하기 전 코드 페이지와 관련된 모든 캐시 라인을 flush 한 다음, TD 종료 후에 캐시 라인들을 전부 reload합니다. Reload 시간이 길다는 것은 TD 내에서 명령어가 실행되었음을 의미합니다. 마지막으로 SEV-step과 유사하게 가상 APIC 타이머 인터럽트를 억제하도록 하이퍼바이저를 수정하여 single-stepping 도중 인터럽트 핸들러가 호출되지 않도록 하였습니다. 

### Adding Spatial Information
Single-stepping으로 얻은 정보만으로는 TD가 현재 어떤 코드를 실행하고 있는지 위치를 알 수 없습니다. 이를 위해 **페이지 폴트 기반 부채널을 결합** 하여 현재 실행 중인 코드 페이지를 파악합니다. SGX나 SEV와 달리 TDX에서는 **TDX module이 TD의 private 메모리 페이지 테이블을 관리** 하기 때문에, 공격자가 직접 접근 권한 비트를 수정하여 페이지 폴트를 강제할 수 없습니다. 대신 TDX module이 제공하는 API를 통해 특정 페이지 접근을 임시로 차단하는 방식을 사용합니다. 

## Attack Primitive Evaluation
