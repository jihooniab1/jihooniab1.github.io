# Summary for TIKTAG: Breaking ARM’s Memory Tagging Extension with Speculative Execution

## Introduction

# TIKTAG: Breaking ARM's Memory Tagging Extension with Speculative Execution 요약

메모리 오염 취약점들은 시스템에 심각한 보안 위협이 되는데, 공격자는 이러한 취약점들을 익스플로잇하여 권한 상승, 정보 유출 등을 달성할 수 있습니다. 이에 대응하여 ARMv8.5-A 아키텍처부터 **Memory Tagging Extension (MTE)** 하드웨어 확장이 제안되었습니다. MTE는 기술적으로 두 가지 하드웨어 연산을 제공합니다.

1. tag: 메모리 주소에 태그를 할당합니다(예: 16바이트 메모리마다 4비트 태그)
2. tag check: 메모리에 접근할 때, 포인터의 태그와 접근할 메모리 주소의 태그를 비교하여 같으면 통과, 다르면 fault

MTE를 활용하면 어떤 태그를 할당하고, 어떤 영역에 태그를 붙이느냐에 따라 다양한 방어 기법을 만들어낼 수 있습니다. 예를 들어 Android Scudo, Chrome PartitionAlloc과 같은 메모리 할당자는 매 할당마다 랜덤 태그를 붙이고, 할당된 메모리를 가리키는 포인터에도 태그가 붙습니다. 포인터가 전파(propagate, 포인터 복사나 함수 인자로 넘기기 등)될 때 태그도 같이 전파됩니다. 동적 할당 메모리에 접근할 때마다 tag check가 강제되고, 이러한 태그는 런타임에 무작위로 할당되기 때문에 공격자가 이를 예측하기 어렵습니다.

MTE는 포인터와 메모리의 태그가 같음을 보장하기 때문에, MTE는 빠르게 메모리 오염을 탐지하고 메모리 오염 취약점을 익스플로잇을 효과적으로 방어할 수 있습니다. CFI 같은 기법은 메모리 오염이 아니라 익스플로잇 단계에서의 제어 흐름 탈취를 탐지한다는 차이가 있습니다.

본 연구는 MTE의 전제를 위협할 수 있는 투기적 실행 공격을 분석하였고 TikTag-v1과 TikTag-v2 두 가젯을 발견하였습니다. 이 가젯들을 활용하면 임의 주소의 MTE 태그를 유출할 수 있습니다. TikTag-v1은 **분기 예측과 data prefetcher의 speculation shrinkage** 를 악용하고, TikTag-v2는 **store-to-load forwarding** 을 악용합니다.

실제 환경에서 공격을 실증하기 위해 MTE 기반 기법이 적용되어 있는 Google Chrome과 Linux Kernel을 대상으로 공격을 수행하였습니다. 실험 결과 TikTag 가젯을 이용하여 짧은 시간 안에 높은 성공률로 태그를 유출할 수 있었습니다. 이후로는 MTE의 장점을 유지하면서 TikTag 가젯을 방어할 수 있는 방어 기법을 제안합니다.

구글의 Project Zero에서는 MTE 메커니즘이 tag check success와 tag check failure 사이에서 구별 가능한 캐시 상태를 만들어내지 않는다는 결론을 내렸고, 그렇기에 투기적 실행으로 인한 태그 유출을 발견하지 못하였습니다. 다른 연구인 StickyTags에서는 TikTag-v1 가젯에 속하는 MTE tag leakage gadget을 발견하였으나, 근본 원인을 spurious tag check fault로 인한 메모리 경합으로 잘못 분석하였습니다. 본 연구에서는 실제 원인이 CPU의 speculation shrinkage와 data prefetch 축소임을 실험으로 규명하였습니다. 이를 통해 random tagging을 유지하면서도 speculation barrier 등으로 방어할 수 있음을 보였으며, 나아가 새로운 TikTag-v2 가젯도 추가로 발견하였습니다.
 
## Background
### Memory Tagging Extension
MTE는 메모리 오염 공격을 막기 위해 제안된 하드웨어 확장으로, ARMv8.5-A 아키텍처부터 도입되었고, 2023년 10월부터 Pixel 8에 채택되었습니다. MTE는 각 16바이트 메모리마다 4비트 태그를 할당하고, 포인터의 사용하지 않는 상위 비트에 그 태그를 저장합니다. 메모리 접근 시 포인터의 태그와 메모리 공간에 할당된 태그가 같은지 비교 후, 같으면 접근을 허용하고 다르다면 CPU가 `tag check fault(TCF)`를 발생시킵니다.

MTE는 synchronous, asynchronous, asymmetric 세 가지 모드를 지원합니다. synchronous 모드는 TCF가 탐지 즉시 동기적으로 발생합니다. asynchronous 모드는 TCF가 컨텍스트 스위치 시점에 비동기적으로 발생합니다. asymmetric 모드는 load 명령어는 synchronous 모드로, store 명령어는 asynchronous 모드로 실행하여 성능과 보안 사이의 균형을 맞춥니다.

`Deterministic tagging`은 할당할 때마다 값이 알려진 태그를 사용하여 메모리 공간을 격리시키거나 공간적 메모리 오염을 탐지할 수 있습니다. `Random tagging`은 할당할 때 생성되는 random tag를 붙입니다. 태그가 4비트이므로 무작위 추측 시 1/16 확률로 태그가 충돌할 수 있어 확률적 방어에 해당합니다. deterministic tagging과 달리 random tagging은 공격자에게 태그 정보를 유출하지 않아 실제 환경에서 널리 활용되고 있습니다.

### Speculative Execution Attack
투기적 실행 공격(speculative execution attack)은 CPU의 투기적 실행을 통해 민감한 정보를 알아내는 공격입니다. Spectre와 Meltdown이 대표적인 예시로, 공격자는 victim 코드를 투기적으로 실행시켜 정상 실행에서는 접근할 수 없는 데이터를 로드합니다. 만약 투기적으로 로드된 데이터가 캐시에 영향을 주면, 공격자는 캐시 상태를 관측하여 데이터를 알아낼 수 있습니다. 보통 이러한 투기적 실행 공격을 수행하려면 공격자가 두 가지를 할 수 있어야 합니다. i) 캐시 상태를 제어하여 victim이 데이터에 접근하기 전에 캐시를 flush하거나 evict, ii) cache hit와 miss를 구별할 수 있는 정밀한 시간 측정 이 가능해야 합니다. 최근에는 투기적 실행을 통해 ASLR, PAC와 같은 하드웨어 보안 기법을 우회하는 공격들이 연구되었습니다.

## Threat Model
공격자 모델에서는 대상 시스템에 MTE가 활성화 되어 있어 타겟 시스템의 각 메모리 할당 때 `random tag`가 붙고, 메모리 접근 때마다 이 태그를 확인합니다. 공격자는 메모리 오염 취약점을 알고 있으며, 이 취약점을 익스플로잇하여 권한 상승을 얻는 것을 목표로 합니다. 공격자 관점에서는 취약점을 트리거하면 높은 확률로 타겟 프로세스가 크래시되면서 관리자가 이를 인지할 수 있는 상황입니다.

## Finding Tag Leakage Gadgets
### Tag Leakage Template
![t1](/assets/img/posts/papers/tiktag_1.png) <br>

먼저 위 그림처럼 공격자가 투기적 실행으로 태그를 유출할 수 있는 speculative MTE tag leakage gadget 템플릿을 설계하였습니다. 핵심 아이디어는 투기적 실행 context에서 MTE tag check를 발생시켜 캐시 상태 변화를 관측하는 것으로, 투기적 실행 도중이므로 mismatch가 발생해도 예외가 발생하지 않습니다. `tag match`와 `tag mismatch` 사이에 차이가 있다면, 공격자는 tag check 결과를 유출하여 태그를 추론할 수 있게 됩니다.

공격자는 `target_addr`에 할당되어 있는 태그인 **$T_m$** 을 알아내려고 하는 상황입니다. 이를 위해 공격자는 `guess_ptr`, `test_ptr` 두 포인터를 준비합니다. `guess_ptr`은 `target_addr`을 가리키면서 **$T_g$** 태그를 달고 있으며, 값은 다음과 같이 구성됩니다.

guess_ptr = (Tg<<56)|(target_addr & ~(0xff<<56))

`test_ptr`은 유효한 태그를 달고 있고, 공격자가 접근할 수 있는 캐시되지 않은 주소를 가리키고 있습니다.

그 후, 공격자는 guess_ptr과 test_ptr로 템플릿을 실행하는데, 템플릿은 `BR, CHECK, TEST` 세 가지 요소로 구성되어 있습니다. BR은 조건 분기로 CHECK와 TEST를 묶어서, **CHECK와 TEST가 투기적으로 실행될 수 있도록** 합니다. CHECK에서는 템플릿이 메모리 명령어 시퀀스를 실행하여 MTE check를 트리거합니다. 그 후 TEST에서는 템플릿이 test_ptr의 캐시 상태를 업데이트하는 명령어를 실행합니다. 가설에 따라 예상되는 흐름은 다음과 같습니다.

공격자는 cond_ptr에 1을 넣고 guess_ptr에도 유효한 주소와 태그를 넣어 branch predictor를 훈련시킵니다. 훈련이 끝난 후, 공격자는 cond_ptr에 0을 넣고 guess_ptr은 target_addr을 가리키게 하고 태그에는 추측된 값을 넣어 템플릿을 실행하는데, 이를 통해 CHECK와 TEST가 투기적으로 실행이 됩니다. 만약 CHECK 단계에서 태그가 일치하면 CPU는 TEST도 투기적으로 실행하여 `test_ptr`에 접근하고 캐시 라인을 채울 것입니다. 만약 일치하지 않으면 CPU는 TEST의 투기적 실행을 중지할 수 있고, 그 결과 캐시 라인이 채워지지 않습니다. 따라서 공격자는 템플릿 실행 후에 **test_ptr 접근 지연을 측정하여 cache hit과 miss를 구분** 하고 tag check 결과를 알아내어 결과적으로 올바른 태그를 알아낼 수 있습니다.

ARMv8.5 기기인 Google Pixel 8, Pixel 8 Pro에서 템플릿을 테스트하였고, CHECK와 TEST에서 사용되는 메모리 명령어들의 개수와 유형도 다양하게 바꿔가며 캐시 상태 변화를 관찰하였습니다. 그 결과 두 가지 MTE leakage 가젯을 식별하였습니다.
