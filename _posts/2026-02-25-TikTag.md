# Summary for TIKTAG: Breaking ARM’s Memory Tagging Extension with Speculative Execution

## Introduction

# TIKTAG: Breaking ARM's Memory Tagging Extension with Speculative Execution 요약

메모리 오염 취약점들은 시스템에 심각한 보안 위협이 되는데, 공격자는 이러한 취약점들을 익스플로잇하여 권한 상승, 정보 유출 등을 달성할 수 있습니다. 이에 대응하여 ARMv8.5-A 아키텍처부터 **Memory Tagging Extension (MTE)** 하드웨어 확장이 제안되었습니다. MTE는 기술적으로 두 가지 하드웨어 연산을 제공합니다.

1. tag: 메모리 주소에 태그를 할당합니다(예: 16바이트 메모리마다 4비트 태그)
2. tag check: 메모리에 접근할 때, 포인터의 태그와 접근할 메모리 주소의 태그를 비교하여 같으면 통과, 다르면 fault

MTE를 활용하면 어떤 태그를 할당하고, 어떤 영역에 태그를 붙이느냐에 따라 다양한 방어 기법을 만들어낼 수 있습니다. 예를 들어 Android Scudo, Chrome PartitionAlloc과 같은 메모리 할당자는 매 할당마다 랜덤 태그를 붙이고, 할당된 메모리를 가리키는 포인터에도 태그가 붙습니다. 포인터가 전파(propagate, 포인터 복사나 함수 인자로 넘기기 등)될 때 태그도 같이 전파됩니다. 동적 할당 메모리에 접근할 때마다 tag check가 강제되고, 이러한 태그는 런타임에 무작위로 할당되기 때문에 공격자가 이를 예측하기 어렵습니다.

MTE는 포인터와 메모리의 태그가 같음을 보장하기 때문에, MTE는 빠르게 메모리 오염을 탐지하고 메모리 오염 취약점을 익스플로잇을 효과적으로 방어할 수 있습니다. CFI 같은 기법은 메모리 오염이 아니라 익스플로잇 단계에서의 제어 흐름 탈취를 탐지한다는 차이가 있습니다.

본 연구는 MTE의 전제를 위협할 수 있는 투기적 실행 공격을 분석하였고 TikTag-v1과 TikTag-v2 두 가젯을 발견하였습니다. 이 가젯들을 활용하면 임의 주소의 MTE 태그를 유출할 수 있습니다. TikTag-v1은 **분기 예측과 data prefetcher의 speculation shrinkage** 를 악용하고, TikTag-v2는 **store-to-load forwarding** 을 악용합니다.

실제 환경에서 공격을 실증하기 위해 MTE 기반 기법이 적용되어 있는 Google Chrome과 Linux Kernel을 대상으로 공격을 수행하였습니다. 실험 결과 TikTag 가젯을 이용하여 짧은 시간 안에 높은 성공률로 태그를 유출할 수 있었습니다. 이후로는 MTE의 장점을 유지하면서 TikTag 가젯을 방어할 수 있는 방어 기법을 제안합니다.

구글의 Project Zero에서는 MTE 메커니즘이 tag check success와 tag check failure 사이에서 구별 가능한 캐시 상태를 만들어내지 않는다는 결론을 내렸고, 그렇기에 투기적 실행으로 인한 태그 유출을 발견하지 못하였습니다. 다른 연구인 StickyTags에서는 TikTag-v1 가젯에 속하는 MTE tag leakage gadget을 발견하였으나, 근본 원인을 spurious tag check fault로 인한 메모리 경합으로 잘못 분석하였습니다. 본 연구에서는 실제 원인이 CPU의 speculation shrinkage와 data prefetch 축소임을 실험으로 규명하였습니다. 이를 통해 random tagging을 유지하면서도 speculation barrier 등으로 방어할 수 있음을 보였으며, 나아가 새로운 TikTag-v2 가젯도 추가로 발견하였습니다.
 
## Background
### Memory Tagging Extension
MTE는 메모리 오염 공격을 막기 위해 제안된 하드웨어 확장으로, ARMv8.5-A 아키텍처부터 도입되었고, 2023년 10월부터 Pixel 8에 채택되었습니다. MTE는 각 16바이트 메모리마다 4비트 태그를 할당하고, 포인터의 사용하지 않는 상위 비트에 그 태그를 저장합니다. 메모리 접근 시 포인터의 태그와 메모리 공간에 할당된 태그가 같은지 비교 후, 같으면 접근을 허용하고 다르다면 CPU가 `tag check fault(TCF)`를 발생시킵니다.

MTE는 synchronous, asynchronous, asymmetric 세 가지 모드를 지원합니다. synchronous 모드는 TCF가 탐지 즉시 동기적으로 발생합니다. asynchronous 모드는 TCF가 컨텍스트 스위치 시점에 비동기적으로 발생합니다. asymmetric 모드는 load 명령어는 synchronous 모드로, store 명령어는 asynchronous 모드로 실행하여 성능과 보안 사이의 균형을 맞춥니다.

`Deterministic tagging`은 할당할 때마다 값이 알려진 태그를 사용하여 메모리 공간을 격리시키거나 공간적 메모리 오염을 탐지할 수 있습니다. `Random tagging`은 할당할 때 생성되는 random tag를 붙입니다. 태그가 4비트이므로 무작위 추측 시 1/16 확률로 태그가 충돌할 수 있어 확률적 방어에 해당합니다. deterministic tagging과 달리 random tagging은 공격자에게 태그 정보를 유출하지 않아 실제 환경에서 널리 활용되고 있습니다.

### Speculative Execution Attack
투기적 실행 공격(speculative execution attack)은 CPU의 투기적 실행을 통해 민감한 정보를 알아내는 공격입니다. Spectre와 Meltdown이 대표적인 예시로, 공격자는 victim 코드를 투기적으로 실행시켜 정상 실행에서는 접근할 수 없는 데이터를 로드합니다. 만약 투기적으로 로드된 데이터가 캐시에 영향을 주면, 공격자는 캐시 상태를 관측하여 데이터를 알아낼 수 있습니다. 보통 이러한 투기적 실행 공격을 수행하려면 공격자가 두 가지를 할 수 있어야 합니다. i) 캐시 상태를 제어하여 victim이 데이터에 접근하기 전에 캐시를 flush하거나 evict, ii) cache hit와 miss를 구별할 수 있는 정밀한 시간 측정 이 가능해야 합니다. 최근에는 투기적 실행을 통해 ASLR, PAC와 같은 하드웨어 보안 기법을 우회하는 공격들이 연구되었습니다.

## Threat Model
공격자 모델에서는 대상 시스템에 MTE가 활성화 되어 있어 타겟 시스템의 각 메모리 할당 때 `random tag`가 붙고, 메모리 접근 때마다 이 태그를 확인합니다. 공격자는 메모리 오염 취약점을 알고 있으며, 이 취약점을 익스플로잇하여 권한 상승을 얻는 것을 목표로 합니다. 공격자 관점에서는 취약점을 트리거하면 높은 확률로 타겟 프로세스가 크래시되면서 관리자가 이를 인지할 수 있는 상황입니다.

## Finding Tag Leakage Gadgets
### Tag Leakage Template
![t1](/assets/img/posts/papers/tiktag_1.png) <br>

먼저 위 그림처럼 공격자가 투기적 실행으로 태그를 유출할 수 있는 speculative MTE tag leakage gadget 템플릿을 설계하였습니다. 핵심 아이디어는 투기적 실행 context에서 MTE tag check를 발생시켜 캐시 상태 변화를 관측하는 것으로, 투기적 실행 도중이므로 mismatch가 발생해도 예외가 발생하지 않습니다. `tag match`와 `tag mismatch` 사이에 차이가 있다면, 공격자는 tag check 결과를 유출하여 태그를 추론할 수 있게 됩니다.

공격자는 `target_addr`에 할당되어 있는 태그인 **$T_m$** 을 알아내려고 하는 상황입니다. 이를 위해 공격자는 `guess_ptr`, `test_ptr` 두 포인터를 준비합니다. `guess_ptr`은 `target_addr`을 가리키면서 **$T_g$** 태그를 달고 있으며, 값은 다음과 같이 구성됩니다.

```
guess_ptr = (Tg<<56)|(target_addr & ~(0xff<<56))
```

`test_ptr`은 유효한 태그를 달고 있고, 공격자가 접근할 수 있는 캐시되지 않은 주소를 가리키고 있습니다.

그 후, 공격자는 guess_ptr과 test_ptr로 템플릿을 실행하는데, 템플릿은 `BR, CHECK, TEST` 세 가지 요소로 구성되어 있습니다. BR은 조건 분기로 CHECK와 TEST를 묶어서, **CHECK와 TEST가 투기적으로 실행될 수 있도록** 합니다. CHECK에서는 템플릿이 메모리 명령어 시퀀스를 실행하여 MTE check를 트리거합니다. 그 후 TEST에서는 템플릿이 test_ptr의 캐시 상태를 업데이트하는 명령어를 실행합니다. 가설에 따라 예상되는 흐름은 다음과 같습니다.

공격자는 cond_ptr에 1을 넣고 guess_ptr에도 유효한 주소와 태그를 넣어 branch predictor를 훈련시킵니다. 훈련이 끝난 후, 공격자는 cond_ptr에 0을 넣고 guess_ptr은 target_addr을 가리키게 하고 태그에는 추측된 값을 넣어 템플릿을 실행하는데, 이를 통해 CHECK와 TEST가 투기적으로 실행이 됩니다. 만약 CHECK 단계에서 태그가 일치하면 CPU는 TEST도 투기적으로 실행하여 `test_ptr`에 접근하고 캐시 라인을 채울 것입니다. 만약 일치하지 않으면 CPU는 TEST의 투기적 실행을 중지할 수 있고, 그 결과 캐시 라인이 채워지지 않습니다. 따라서 공격자는 템플릿 실행 후에 **test_ptr 접근 지연을 측정하여 cache hit과 miss를 구분** 하고 tag check 결과를 알아내어 결과적으로 올바른 태그를 알아낼 수 있습니다.

ARMv8.5 기기인 Google Pixel 8, Pixel 8 Pro에서 템플릿을 테스트하였고, CHECK와 TEST에서 사용되는 메모리 명령어들의 개수와 유형도 다양하게 바꿔가며 캐시 상태 변화를 관찰하였습니다. 그 결과 두 가지 MTE leakage 가젯을 식별하였습니다.

### Tag Leakage Fuzzing
MTE tag leakage 가젯을 자동으로 발견하기 위해 Spectre-v1 퍼저와 유사한 방식으로 퍼저를 개발하였습니다. 퍼저는 먼저 cond_ptr, guess_ptr, test_ptr에 메모리를 할당한 다음 cond_ptr과 guess_ptr은 각각 고정된 128바이트 메모리 영역을 가리키게 하고, test_ptr은 랜덤 값으로 초기화된 4KB 메모리 영역 내의 가변 128바이트 정렬 주소를 가리키게 합니다. 그 다음 사용 가능한 레지스터(x0-x28) 중에서 무작위로 두 개를 골라 cond_ptr과 guess_ptr을 할당합니다. 나머지 레지스터들은 4KB 메모리 영역 내의 128바이트 정렬 주소나 랜덤 값을 담습니다.

그 후 퍼저는 사전에 정의된 instruction들(ldr, str, eor, orr, nop, isb 등)로 CHECK, TEST 블록을 채웁니다. 초기 테스트 케이스가 주어졌을 때, 퍼저는 instruction을 삽입, 제거, 교체하여 새로운 테스트 케이스를 생성합니다.

테스트 케이스는 두 페이즈에 걸쳐서 실행이 됩니다. i) branch training phase(cond_ptr이 true, guess_ptr도 correct tag), ii) speculative execution phase(cond_ptr이 false, guess_ptr은 correct or wrong tag). 퍼저는 각 테스트 케이스를 두 번씩 실행합니다.

첫 번째 실행은 branch training을 한 다음 올바른 태그로 투기적 실행을 합니다. 두 번째 실행은 training을 한 다음 틀린 태그로 투기적 실행을 합니다. 각 실행 후 접근 지연을 측정하여 두 실행 사이의 캐시 상태를 비교합니다. 이 과정은 4KB 메모리 영역의 각 캐시 라인에 대하여 반복되고, 만약 눈에 띄는 차이가 발견되면 잠재적 가젯으로 고려합니다.

## TikTag Gadgets
### TikTag-v1: Exploiting Speculation Shrinkage
![t2](/assets/img/posts/papers/tiktag_2.png) <br>

실험 결과 MTE 태그를 유출하려면 템플릿이 다음 조건들을 만족해야 하는 것을 발견하였습니다. **i) CHECK는 최소한 두 개 이상의 guess_ptr load를 포함해야 합니다. tag check를 한 번만 하면 test_ptr이 항상 캐시가 됩니다. ii) TEST는 test_ptr에 접근해야 합니다. iii) CHECK는 BR과 5 CPU cycle 이내로 가까워야 하고, TEST는 CHECK로부터 10 cycle 이상 떨어져 있어야 합니다.**

위 그림은 TikTag-v1 가젯을 나타내고 있습니다. CPU는 BR 단계에서 분기를 잘못 예측하여 CHECK를 투기적으로 실행합니다. CHECK에서 guess_ptr은 2차례 이상 역참조 되어 $T_g$ 태그를 $T_m$과 비교하는 tag check를 트리거합니다. GAP은 BR과 TEST 사이의 시간 갭을 만듭니다. 10 cycle 이상의 갭만 만들 수 있으면 어떤 명령어든 상관 없습니다. TEST에서는 test_ptr이 접근됩니다.

실험 결과 TikTag-v1이 ARM Cortex-X3을 대상으로 MTE synchronous, asynchronous 두 모드에서 태그를 효과적으로 추출해내는 것을 확인하였습니다. 물리 CPU cycle counter(PMCCNTR_EL0)를 활용하여 시간을 측정하고 L1 캐시 hit 여부를 결정하였습니다. 실제 환경에서는 유저 공간에서 사용 가능하지만 해상도가 낮은 가상 CPU cycle counter(CNTVCT_EL0)를 사용하였으나, 충분히 태그를 추출할 수 있었습니다. <br>

![t3](/assets/img/posts/papers/tiktag_3.png) <br>

TikTag-v1을 실행한 후 cache hit 비율을 측정하기 위해 i) CHECK에서 load의 개수, ii) TEST에서 메모리 접근의 유형, iii) GAP을 orr(bitwise OR) 명령어로 채운 다음 GAP의 길이를 바꿔가며 실험을 진행하였습니다. 결과는 위와 같습니다. x축은 `Len(GAP)`을, y축은 1000회 실행 후의 test_ptr 캐시 hit 비율을 나타냅니다.

Len(CHECK)=1일 때는 태그 match와 mismatch 사이의 차이가 없었습니다. Len(CHECK)=2일 때는 tag match일 때는 항상 캐시되었으나 mismatch일 때는 hit rate가 급감하였습니다. 이러한 변화는 모든 TEST 메모리 접근 타입에서 관찰되었고, GAP을 10개 이상의 orr로 채웠을 때 차이가 발생하였습니다.

태그가 불일치할 때는 test_ptr 캐시 히트가 주기적으로 발생하였고, Len(CHECK)가 늘어날수록 주기가 짧아졌습니다. 2였을 때는 6번 중 5번이 hit(화살표 a), 4일 때는 3번 중 2번이 hit(화살표 b), 8일 때는 store 접근은 3번에 1번 hit(화살표 c), load 접근은 처음에는 항상 hit였다가 점점 2번 중 1번 hit로 변화하였습니다(화살표 d). <br>

![t4](/assets/img/posts/papers/tiktag_4.png) <br>

가젯을 분석한 결과 tag check가 **CPU의 data prefetching과 투기적 실행**에 영향을 주는 것을 발견하였습니다. 일반적으로 최신 CPU의 투기적 메모리 접근은 두 유형으로 나뉩니다: 투기적 실행, data prefetching. TikTag-v1의 root cause를 알아내기 위해 위와 같은 제거 연구(ablation study)를 진행하였습니다. 먼저 투기적 실행의 효과를 없애기 위해 CHECK와 TEST 사이에 speculation barrier(sb)를 삽입하였고, data prefetching의 효과를 제거하기 위해 분기 훈련과 투기적 실행 phase에서 test_ptr이 가리키는 주소를 다르게 하여 CPU prefetcher가 올바른 주소를 예측하지 못하게 하였습니다.

`Baseline`에는 speculation barrier도 없고, 분기 훈련과 투기적 실행 phase 모두 같은 주소에 접근합니다. `-SE`의 경우 CHECK와 TEST 사이에 barrier를 삽입하여 TEST가 투기적으로 실행되지 않도록 하였는데, 동일한 캐시 상태 차이가 관찰되었습니다. `-SE & -DP`의 경우 speculation barrier에 더해 test_ptr의 주소도 다르게 하여 data prefetching도 제거하였습니다. 이 경우 test_ptr은 항상 캐시되지 않았으며, `-SE`와 `-SE & -DP`를 비교하면 data prefetching이 활성화된 경우(-SE)에만 캐시 상태 차이가 발생하므로, tag mismatch 시 CPU가 data prefetching을 중단하는 것이 하나의 root cause임을 확인하였습니다. 마지막으로 `-DP`의 경우 speculation barrier를 제거하여 투기적 실행은 허용하면서 test_ptr의 주소만 다르게 하였는데, 이 경우에도 캐시 상태 차이가 관찰되었습니다. `-DP`와 `-SE & -DP`를 비교하면 data prefetching이 없는 상황에서도 투기적 실행만으로 캐시 상태 차이가 발생하므로, 투기적 실행 역시 root cause에 포함됨을 확인하였습니다.

본 연구에서는 CPU가 tag check fault 발생 시 투기적 실행과 data prefetching을 중단하여 최적화를 시도하는 것이라 추측하였습니다. 분기 예측 실패를 감지하면, CPU는 잘못된 분기로의 투기적 실행과 data prefetch를 중단하여 recovery time을 절약할 수 있습니다. 또한 CPU가 일정 길이(5 cycle로 추정)의 window를 유지하며 잘못된 분기 진행을 감지하는 것으로 추측하였습니다. 또한 TEST에서 store 접근을 할 때 speculation barrier는 test_ptr을 항상 캐시되지 않도록 하는데, 이를 통해 CPU가 store 접근에 대해서는 prefetch를 하지 않는 것을 확인하였습니다. <br>

![t5](/assets/img/posts/papers/tiktag_5.png) <br>

퍼징을 수행하며 위와 같이 linked list traversal을 활용하는 변형 가젯을 발견하였습니다. 가젯 실행 전에 4개의 노드로 구성된 링크드 리스트를 초기화하는데, 각 노드는 다음 노드를 가리키며(ptr0 → ptr1 → ptr2 → ptr3), 마지막 노드인 ptr3의 캐시 라인은 flush됩니다. 가젯은 ptr0부터 ptr3까지 순서대로 링크드 리스트를 순회하며, TEST는 분기 결과가 참일 때만 ptr3에 접근합니다. 가젯 실행 후 ptr3의 캐시 hit 비율을 측정합니다.

위 그림 왼쪽의 variant 1은 TEST가 조건 분기 BR의 true branch 안에 위치하고, variant 2는 TEST가 조건 분기 밖에 위치하여 true, false branch가 합쳐지는 지점 이후에 실행됩니다. 두 변형 모두 태그가 일치하면 ptr3이 캐시되고, 불일치하면 캐시되지 않는 기존 TikTag-v1과 동일한 동작을 보였습니다. 이는 linked list traversal과 같은 현실적인 코드 패턴에서도 태그 유출이 가능하며, 조건 분기 범위 밖의 메모리 접근에서도 태그 유출이 관찰될 수 있음을 의미합니다.

microarchitectural 레벨에서 TikTag을 방지하려면 tag check fault가 발생할 때 투기적 실행이나 data prefetching에 변화를 주어서는 안 됩니다. 소프트웨어 레벨에서 방지하려면 다음 두 가지 접근이 가능합니다.

1. Speculation barrier: TEST가 store 접근을 한다면 TEST 전에 barrier나 `isb` 같은 명령어 동기화 instruction을 삽입하여 항상 캐시가 안 되도록 할 수 있습니다. load 접근의 경우 CHECK 전에 barrier을 배치하여 tag check fault가 투기적으로 발생하지 않도록 합니다.

2. Padding instructions: CHECK 전에 명령어 시퀀스를 넣어 타임 윈도우를 늘리면, CPU는 투기적 실행을 축소시키는 일이 없어지고 test_ptr이 항상 캐시됩니다. 

### TikTag-v2: Exploiting Store-to-Load Forwarding
Spectre-v4, LVI 공격 사례에서 영감을 받아, 본 연구에서는 store-to-load forwarding을 트리거하여 태그 유출을 실험하였습니다. 실험 결과, 다음 조건을 만족할 때 tag check 결과에 따라 store-to-load forwarding 결과가 변한다는 것을 발견하였습니다: i) CHECK가 store-to-load forwarding을 트리거하고, ii) TEST는 forwarded value에 의존적인 메모리에 접근해야 합니다. 만약 CHECK에서 tag가 일치하면 TEST의 메모리 접근이 캐시되고, 일치하지 않으면 캐시되지 않습니다. <br>

![t6](/assets/img/posts/papers/tiktag_6.png) <br>

위 그림은 TikTag-v2 가젯을 나타내고 있습니다. CPU가 BR에서 분기를 잘못 예측하고 투기적으로 CHECK, TEST를 실행하는 것은 똑같습니다. CHECK 단계에서 CPU는 test_ptr에 store하고 guess_ptr에서 바로 그 값을 로드하여 store-to-load forwarding을 트리거합니다. 태그가 일치하면 CPU는 test_ptr를 forwarding 하고 TEST 단계에서 투기적으로 test_ptr에 접근합니다. CHECK 단계에서 tag check가 실패하면 CPU는 test_ptr이 forwarding 되는 것을 막고 test_ptr에 접근하지 않습니다. <br>

![t7](/assets/img/posts/papers/tiktag_7.png) <br>

실험 결과 TikTag-v2에 필요한 한 가지 조건을 식별할 수 있었는데, CHECK의 store, load 명령어들이 5 instructions 이내에 실행되어야 한다는 것입니다. 이 조건을 만족하면 위 그래프와 같이 tag match/mismatch 사이의 뚜렷한 변화가 드러납니다. 

이를 검증하기 위해 위 어셈블리와 같이 SLIDE와 GAP 두 명령어 시퀀스를 삽입하여 실험을 수행하였습니다. SLIDE는 가젯 앞에 삽입되어 CPU 파이프라인에서 store-to-load forwarding의 정렬을 제어하고, GAP은 CHECK의 store와 load 명령어 사이에 삽입되어 둘 사이의 거리를 제어합니다. 두 시퀀스 모두 이전 명령어에 의존하는 orr 명령어로 구성되어 레지스터나 메모리 상태에는 영향을 주지 않습니다. SLIDE를 0부터 40까지 GAP을 0부터 4까지 변화시켜가며 실험한 결과는 위와 같습니다. x축은 GAP의 길이, y축은 1000회 실행 중 cache hit 비율을 나타낸 것입니다.

tag match일 때는 모든 조건에서 test_ptr의 캐시 hit 비율이 거의 100%였습니다. tag mismatch일 때는 SLIDE 길이가 5의 배수가 될 때마다 hit rate가 감소하였습니다. Len(GAP)=0일 때 hit rate 감소가 가장 자주(5회 중 4회) 발생하였고, Len(GAP)이 1, 2, 3으로 늘어날수록 감소 횟수가 줄어들었으며, Len(GAP)=4일 때는 hit rate 감소가 전혀 관찰되지 않았습니다.

본 연구에서는 TikTag-v2의 root cause를 CPU가 tag check fault 발생 시 store-to-load forwarding을 차단하는 것 때문으로 추측하였습니다. CPU는 Load-Store Queue(LSQ)와 같은 내부 버퍼를 활용하여 store-to-load 의존성을 감지하고 데이터를 forwarding합니다. ARM의 관련 특허에 따르면 store-to-load 의존성이 감지되면 load 명령어가 tag check를 건너뛰고 항상 데이터를 forwarding할 수 있습니다. 이 경우 Len(GAP)이 4 이상일 때처럼 store와 load가 다른 사이클에 실행되어 의존성이 감지되면 tag check 결과와 무관하게 항상 forwarding이 성공합니다. 그러나 Len(GAP)이 4 미만일 때는 store와 load가 같은 사이클에 실행되어 의존성을 감지하지 못하고 load에 대해 tag check를 수행하게 되며, tag check fault 발생 시 forwarding이 차단됩니다. SLIDE 길이가 5의 배수마다 hit rate가 감소하는 것은 Cortex-A715가 한 사이클에 5개의 명령어를 처리하는 것과 관련이 있습니다.

Micro-architectural 레벨에서 TikTag-v2를 막으려면 CPU는 tag check 결과에 관계없이 항상 store-to-load forwarding을 허용하거나 막도록 설계되어야 합니다. 항상 막으면 성능 저하가 발생할 수 있으니 항상 허용하는 것이 효율적인 해결책이 될 수 있습니다. 소프트웨어 레벨에서는 다음과 같은 방어책들이 있습니다.

1. Speculation barrier: CHECK 전에 barrier가 삽입되면 CPU가 store-to-load를 투기적으로 실행하지 않아 test_ptr도 캐시되지 않습니다.
2. Preventing Gadget Construction: CHECK의 store과 load 명령어가 5 instruction 이내에 실행되지 않으면 test_ptr은 항상 캐시가 될텐데, 잠재적인 가젯들에 nop 같은 dummy instruction을 삽입하거나 명령어 순서를 재배치하여 가젯을 수정할 수 있습니다. 

## Real-World Attacks
TikTag의 위력을 입증하기 위해 Chrome, Linux Kernel에 대한 공격을 수행하였습니다. TikTag 가젯을 사용하려면 먼저 가젯이 타겟의 주소 공간에서 실행되어야 하고, 공격자가 캐시 상태를 제어하고 관찰할 수 있어야 합니다.

### Attacking Chrome Browser
웹 브라우저는 JavaScriptsㅏ HTML 같은 신뢰할 수 없는 웹 컨텐츠를 처리하기 때문에 웹 기반 공격의 주된 공격 벡터입니다. 

#### Threat Model
공격자는 렌더러 프로세스의 메모리 오염 취약점을 익스플로잇을 시도하려고 하고, victim은 공격자가 제어하는 악성 웹사이트를 방문하는 상황을 가정합니다. 웹페이지에는 조작된 HTML, JavaScript가 있고 victim의 렌더러 프로세스의 메모리 오염 취약점을 익스플로잇합니다. ASLR, CFI, site isolation, V8 sandbox와 같은 Chrome의 최신 방어기법들이 전부 적용되어 있고 PartitionAlloc에서 랜덤 MTE를 활성화하고 있습니다. 

#### Constructing TikTag Gadget
V8 JavaScript 환경에서 TikTag-v2를 성공적으로 구축하였고 태그를 유출할 수 있었습니다. v1 가젯은 V8 sandbox escape 과정에서 BR과 CHECK의 간격이 너무 벌어져서 가젯 구조상의 이유로 구축할 수 없었습니다. <br>
